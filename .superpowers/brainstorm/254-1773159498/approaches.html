Architecture Approaches

How should we structure the prototype?

A

Monolithic Single-Page App

One React app with all logic inline — chat state machine, typewriter animation, visualization renderer, payload switching. Simple Vite setup, minimal files.

Pros

  • Fastest to build — everything in one place
  • No component architecture overhead
  • Easy for Sean to read top-to-bottom

Cons

  • Gets messy fast if you iterate
  • Hard to reuse visualization logic from the real MCP app
  • Mixing concerns (chat + viz + animation)
B

Component-Based React App

Clean separation: ChatPanel, VisualizationPanel, SimulationEngine. Shared types imported from the existing MCP app's schema. Tailwind or CSS modules for styling.

Pros

  • Clean architecture — each piece is testable and replaceable
  • Shares types with the real MCP app (FormulaPayload)
  • Natural path to swap simulation for real API later
  • Sean can focus on payload creation without touching UI code

Cons

  • More files and structure upfront
  • Slightly more complex for a "prototype"
C

Iframe Embedding

Chat panel in the main React app, but the visualization panel embeds the actual MCP app HTML via iframe, communicating via postMessage. Maximum reuse of existing code.

Pros

  • Real MCP app rendering — zero duplication
  • Visualization is always in sync with the actual product

Cons

  • iframe comms are finicky and hard to debug
  • Styling isolation can cause visual inconsistencies
  • Adds complexity for a prototype
  • Harder for Sean to understand the full picture

💡 Recommendation: Option B

Component-based gives us the best balance — clean enough for Sean to grok the concept, structured enough to iterate on, and naturally aligned with the real MCP app's data model. The "slightly more files" concern is minimal with Vite's scaffolding.
